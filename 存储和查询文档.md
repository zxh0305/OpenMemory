# OpenMemory 存储和查询文档

## 1. 当前存储逻辑

当前系统会把一次输入拆成两部分存储：

- 原始记忆：存入 `raw_memory_inputs` 表
- 提取后的记忆：存入 `memories` 表

关系说明：

- `raw_memory_inputs.id` 是原始输入记录 ID
- `memories.metadata.raw_record_id` 用来关联对应的原始输入

也就是说：

- 查原文，看 `raw_memory_inputs`
- 查结构化记忆，看 `memories`

补充说明：

- 普通短文本：`memories` 中通常只会有原子事实
- 长文本：`memories` 中会有两类内容
  - 原子事实 `fact`
  - 整段摘要 `session_summary`
- 长文本的分段摘要只保留在 `raw_memory_inputs.summary` 中，不单独写入 `memories`
- 如果请求带了 `metadata.sync_time`，写入 `memories` 的内容会自动带日期前缀
  - 例如：`3.10日用户提到充电站供电可能不稳定`
  - 例如：`3.10日摘要：用户表达了对充电站供电问题的担忧...`

## 2. 存储记忆

### 2.1 开启 AI 提取

开启后：

- 原文会存入 `raw_memory_inputs`
- AI 会提取原子事实
- 如果是长文本，还会额外生成一条整段摘要
- 最终写入 `memories` 的是：
  - 原子事实
  - 长文本整段摘要

```bash
curl -s --http1.1 --max-time 180 \
  -X POST http://127.0.0.1:8765/api/v1/memories/ \
  -H "Content-Type: application/json" \
  -d '{
    "user_id":"user",
    "text":"我叫张三，今年35岁了，我妻子叫李思，今年33岁，儿子今年刚上幼儿园",
    "metadata": {},
    "infer": true,
    "app":"openmemory"
  }'
```

### 2.2 长文本存储（带时间信息）

如果是长文本，建议不要手拼超长 JSON，而是从文本文件生成请求体。

假设原始文本文件是：

`/Users/zhangxianghao/Documents/资料/测试数据/id=3三月十日文本.txt`

可直接这样发：

```bash
jq -Rs \
  --arg user_id "3" \
  --arg app "openmemory" \
  --arg source "gosexm_chat" \
  --arg sync_time "2026-03-10T19:58:44Z" \
  '{
    user_id: $user_id,
    text: .,
    metadata: {
      source: $source,
      sync_time: $sync_time
    },
    infer: true,
    app: $app
  }' /Users/zhangxianghao/Documents/资料/测试数据/id=3三月十日文本.txt \
| curl -s --http1.1 --max-time 180 \
    -X POST http://127.0.0.1:8765/api/v1/memories/ \
    -H "Content-Type: application/json" \
    -d @-
```

长文本写入后，`memories` 中通常会看到：

- `3.10日摘要：...`
- `3.10日用户提到...`

### 2.3 不开启 AI 提取

关闭后：

- 原文仍然会存入 `raw_memory_inputs`
- 原文会作为一条记忆直接存入 `memories`

```bash
curl -s --http1.1 --max-time 180 \
  -X POST http://127.0.0.1:8765/api/v1/memories/ \
  -H "Content-Type: application/json" \
  -d '{
    "user_id":"user",
    "text":"我叫张三，今年35岁了，我妻子叫李思，今年33岁，儿子今年刚上幼儿园",
    "metadata": {},
    "infer": false,
    "app":"openmemory"
  }'
```

## 3. 返回字段说明

存储接口返回中常用字段：

- `raw_record_id`：原始输入表中的记录 ID
- `summary`：原始输入的摘要
- `extracted_facts`：提取出的事实列表
- `created_count`：写入记忆表的条数
- `created_memories`：实际写入的记忆列表

对于长文本：

- `summary`：会包含整段摘要以及分段摘要拼接内容
- `extracted_facts`：只包含最终保留下来的原子事实，不包含整段摘要
- `created_memories`：会同时包含
  - `fact`
  - `session_summary`

## 4. 查询原始记忆

### 4.1 查询原始输入列表

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/raw-inputs?user_id=user"
```

### 4.2 按应用筛选原始输入

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/raw-inputs?user_id=user&app=openmemory"
```

### 4.3 查询某条原始输入详情

这里的 `raw_record_id` 来自存储接口返回值。

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/raw-inputs/你的raw_record_id"
```

这个接口会返回：

- `original_text`
- `summary`
- `extracted_facts`
- `linked_memories`

## 5. 查询提取后的记忆

### 5.1 查询记忆列表

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/?user_id=user"
```

如果你的用户是 `3`：

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/?user_id=3"
```

### 5.2 按关键词查询记忆

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/?user_id=user&search_query=张三"
```

例如查询长文本里和“充电站”有关的记忆：

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/?user_id=3&search_query=充电站"
```

### 5.3 查询某条记忆详情

```bash
curl -s --http1.1 \
  "http://127.0.0.1:8765/api/v1/memories/你的memory_id"
```

## 6. 直接查看数据库

### 6.1 查看原始输入表

```bash
docker compose exec -T openmemory-api sqlite3 openmemory.db \
  "select id, original_text, summary, extracted_facts, processing_status, created_at from raw_memory_inputs;"
```

### 6.2 查看提取后的记忆表

```bash
docker compose exec -T openmemory-api sqlite3 openmemory.db \
  "select id, content, metadata, state, created_at from memories;"
```

如果你想区分“摘要记忆”和“原子事实记忆”，可以这样查：

```bash
docker compose exec -T openmemory-api sqlite3 openmemory.db \
  "select content, json_extract(metadata, '$.fact_type') as fact_type, json_extract(metadata, '$.source_app') as source_app from memories order by created_at desc;"
```
